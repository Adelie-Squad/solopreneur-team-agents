# v1.2.2 — 용어/디렉토리 구조 명확화

> Product/Project/Repository 용어 혼동과 Workspace ↔ Product-Repos 경로 관계에 대한 사용자 혼동을 해소한다. 기능 변화보다 **setup UX**와 **문서 명료화**가 주 축.

**기준 버전:** v1.1.5 (최신 퍼블리시)
**분류:** UX / DX 개선
**심각도:** Medium — 기능 장애는 아니지만 첫 사용자가 구조를 오해해 잘못된 위치에 제품을 만들거나, GitHub repo와 SoloSquad "project"를 동일시하는 혼동이 반복 발생 예상

---

## 1. 배경 — 사용자 피드백

초기 사용자(본인 QA 과정)에서 다음 두 질문이 제기됨:

**Q1.** 프로젝트는 리포지토리 단위인가? `bv-ai-native-po`를 만들었으면 `projects/` 하위에 리포지토리를 클론하면 되는가? 그럼 `bv-ai-native-po`는 깃허브로 치면 organization인가? 그럼 setup 시 안내가 나와야 하는 것 아닌가?

**Q2.** `bv-ai-native-po`는 `my-workspace`의 하위인가? 하위라면 workspace 아래에 제품을 만들도록 강제해야 하지 않나? (개발 프로젝트는 보통 `Documents` 하위에 만들지 않는 것처럼.) 무관한 경로나 하위 경로에 만들면 어떤 문제가 생기나?

두 질문 모두 **설계 의도는 있으나 사용자가 그 의도를 읽어낼 힌트가 setup/문서에 부족**해서 발생. 즉 코드 문제가 아니라 전달 문제.

## 2. 현 설계 정리

### 2.1 3-Layer Context — 설계 의도

```
Layer 0 (Universal)  = my-workspace/core/        오너 프로필, 원칙, 글쓰기 스타일
Layer 1 (Product)    = <REPOS_BASE_PATH>/<slug>/ 제품 브리프, 메모리(JSONL), 플랫폼 매핑
Layer 2 (Project)    = <...>/<slug>/projects/    워크플로우별 격리된 산출물 폴더
```

### 2.2 용어 실체

| 개념 | 정체 | 구현 근거 | 예시 |
|---|---|---|---|
| **Workspace** | SoloSquad 설정 중앙. `solosquad init`을 실행한 위치 | `src/util/paths.ts:20` `getWorkspaceDir() = process.cwd()` | `C:\Dev\my-workspace` |
| **Product** | 제품(혹은 스타트업/팀) 하나. 독립 메모리·컨텍스트 보유 | `src/util/config.ts:7-11` `Product = { name, slug, github_org? }` | `BV AI Native PO` → slug `bv-ai-native-po` |
| **Project** | 그 제품 안의 한 개 **워크플로우 인스턴스**(PMF 검증 / 기능 출시 / 리브랜딩) | `assets/orchestrator/SKILL.md:287-318` | `projects/jeonse-guard/stage-1-research/...` |
| **Repository** | 진짜 코드 리포지토리. **SoloSquad의 1급 개념이 아님** | — | GitHub `org/repo` — 별도로 clone하여 사용 |

### 2.3 Product 디렉토리 기본 위치

`src/util/platform.ts:36-41`:

```ts
export function defaultReposPath(): string {
  if (IS_WINDOWS) {
    return path.join(os.homedir(), "Documents", "solosquad-repos");
  }
  return path.join(os.homedir(), "repos");
}
```

즉 기본값:
- **macOS/Linux**: `~/repos/<slug>`
- **Windows**: `~/Documents/solosquad-repos/<slug>`

Workspace와는 **독립 트리**. 사용자가 init 프롬프트에서 `./repos`를 입력하면 workspace 하위로 들어가지만 그건 **명시적 옵션**일 뿐 기본이 아님.

## 3. 혼동 지점 분석

### 3.1 Q1 — Project ≠ Repository 혼동

**원인:**
- 소프트웨어 세계에서 "project"는 통상 코드 리포지토리와 동치로 쓰임 (IDE 용어, JetBrains/VS Code "Open Project" 등)
- SoloSquad의 `project`는 orchestrator가 관리하는 **한 개 워크플로우**(PRD + 단계별 산출물 + 세션 컨텍스트)
- `projects/{project_id}/` 아래가 비어있는 초기 상태에서 "여기에 repo를 clone하나?"가 자연스러운 오해

**`init` 시 실제 프롬프트:**
```
Product/Organization name:  ← 설명 없음
GitHub org (Enter to skip): ← 왜 org인지 모름, 채워야 할 이유도 불명
```

**문서상 부재:**
- README.md, setup-guide.md 어디에도 "Product ≠ Repository" "Project ≠ Repository" 명시 없음
- 코드 리포지토리를 따로 clone해야 한다는 가이드 없음 → 사용자는 `projects/` 안에 clone해야 하나 고민

### 3.2 Q2 — Workspace와 Product Repos 경로 분리의 동기가 불투명

**원인:**
- `REPOS_BASE_PATH`가 별도 존재하는 이유(여러 workspace가 같은 product를 공유 가능, workspace 삭제 시 product 데이터 보존 등)가 문서에 없음
- Windows 기본값 `~/Documents/solosquad-repos`는 개발자 관례(`~/Code/`, `~/repos/`)와 어긋나 어색함
- `solosquad status`에 두 경로가 함께 표시되지 않음 → 사용자가 전체 레이아웃을 인지하기 어려움

**실제 상황:**
- Workspace: `C:\Dev\my-workspace\`
- Product: `C:\Users\...\Documents\solosquad-repos\bv-ai-native-po\`

이 두 경로가 따로 떨어진 이유를 사용자가 추론할 방법이 없음.

### 3.3 혼동의 2차 효과

- 사용자가 product 디렉토리를 임의로 옮기면 `.env`의 `REPOS_BASE_PATH`가 맞지 않아 봇이 실패 — 에러 메시지는 친절하지 않을 가능성
- Docker 사용 시 `${REPOS_BASE_PATH:-./repos}:/repos` 마운트가 어떻게 동작하는지 이해 못 하면 "왜 컨테이너가 제품 디렉토리를 못 봐요?"로 이어질 수 있음

## 8. 후속 고려

- Phase 4에서 Windows 기본값을 바꿀 경우: 기존 사용자 마이그레이션 도구 필요 여부 검토 (자동 이전 또는 `solosquad migrate` 같은 1회성 명령)
- 향후 "Repository" 개념을 1급으로 격상할지 별도 논의 — 제품 안의 코드 repo를 SoloSquad가 추적/연동하면 배포/CI까지 확장 가능. 다만 범위 폭발 우려.
- `projects/` 하위에 orchestrator가 자동으로 첫 프로젝트(예: `pmf-discovery`)를 만들어 제품 등록 후 즉시 워크플로우가 비어있지 않도록 하는 옵션 검토.
